01
Inventory sync: MYOB to Tradify
Item catalogue and sell prices pushed nightly from MYOB into Tradify
What it does

MYOB is master. One-way push with no sync loops and no feedback from Tradify back to MYOB. Each run pulls the item catalogue from MYOB and creates or updates items in Tradify by match key. Run it twice and you get the same result.

Scope
  • Item catalogue and sell prices only
  • Stock-on-hand deferred, but architecture allows it to be added later without a rebuild
  • No purchase prices, supplier costs, or pricing tiers in scope
Match key and schedule
  • MYOB Item Number matches to Tradify item code
  • First run is a bulk load (Tradify starts empty on items)
  • Subsequent runs: create or update only
  • Nightly scheduled run, plus a manual trigger for immediate pushes
Open dependency

Tradify must support programmatic item creation and update. Confirm against live key before build. If limited or absent, scope changes significantly.

Verification

No duplicatesRe-run produces updates only, no duplicate items created

02
Webform to Tradify Enquiry
Public form submissions routed into Tradify as leads for the office to triage
What it does

Public webform submissions land as Enquiries in Tradify, keeping the main job board clean. Office staff triage and convert to jobs manually. The conversion decision stays with the business.

Form structure
  • Top-level branch: Appliance Repair or Electrical Work, each with its own question set
  • Both paths collect contact details, service address, problem description, photos, urgency, and preferred time
  • Custom-built (client preference over a third-party form builder)
  • Embeds in existing site and posts to the integration layer
What the system does with it
  • Receives the submission including photos
  • Creates the Tradify Enquiry with all submitted data and attachments
  • Spam protection required (public-facing form)
  • Photo storage solution needed alongside attachment to Enquiry
Open dependency

Tradify must support Enquiry creation and file attachment. If not: fallback to Customer and note, or job with a web lead tag. Confirm against live key before build.

Verification
End to endSubmit form — Enquiry appears in Tradify with all fields and photos
Both pathsAppliance Repair and Electrical Work funnels tested independently
RejectionSpam and invalid input correctly rejected before submission
03
Automated customer updates
Job status changes trigger customer-facing SMS and email, plus internal notifications for parts procurement
Trigger milestones
  • Booked and scheduled
  • On the way today
  • Parts ordered, with ETA (appliance repair specific)
  • Parts arrived, repair booked to be started on [date]
  • Job completed
  • Invoice sent / payment due
Channel logic

Time-sensitive updates (e.g. "on the way") go as SMS. Informational updates (e.g. "parts ordered") go as email. Each status maps to a channel and a message template. Some templates reference fields the office enters on the job, such as a parts ETA date.

Internal notifications

Alongside customer-facing messages, Bainsy needs to be notified when jobs reach milestones that affect parts — particularly when a job is booked, when parts are ordered, and when a repair is scheduled. This runs on the same trigger system but routes to an internal channel (email or SMS) rather than to the customer.

Job board visibility

Notifications alone are not enough. Bainsy also needs a live view of active jobs so he can see what is coming, plan procurement, and make pricing decisions. Whether this is a Tradify view, a filtered dashboard, or something else needs to be confirmed — but it is a named requirement for Part 4 to function correctly.

Detection

Tradify webhooks if available; polling as the safe fallback. To be confirmed against the live system before build.

Requirements
  • SMS provider with AU sender number (Twilio or MessageMedia; cost per message is the deciding factor)
  • Email sender configured
Verification
Per milestoneTest job moved through each status — correct message, correct channel, correct variables
No duplicatesSystem prevents repeat sends on the same trigger
04
Invoicing
Depends on Part 1 — invoicing in Tradify requires the item catalogue to be live first
Current state

Invoicing is not currently done inside Tradify. Without a full item catalogue — part numbers, descriptions, and costs — Tradify cannot generate meaningful job invoices. Part 4 cannot be built until Part 1 is in place and the catalogue is populated.

Flow (once Part 1 is live)

Tradify (completed job with logged labour and parts) to MYOB (invoice drafted and reviewed) to customer. The native connection between Tradify and MYOB is designed to handle this. Tradify creates draft invoices from completed jobs; the draft, review, and send step in MYOB provides the internal check the business needs.

Bainsy's role

Bainsy manages parts procurement and pricing. He needs both job notifications (covered in Part 3) and a live view of the job board so he can see what is coming, plan procurement, and make pricing decisions. He also defines which parts are overdue and what discount applies — those decisions need to flow through to item costs in MYOB before invoices are generated.

Overdue stock

Bainsy determines when a part is overdue and what it should be discounted to. That decision needs to flow through to the item cost in MYOB so invoices reflect the right price. The question for build is whether Bainsy updates costs directly in MYOB, or whether there's a simpler way to flag and apply discounts at invoice time.

Dependencies
  • Part 1 must be complete and the item catalogue live in Tradify before this part can be scoped or built
  • Bainsy needs a reliable way to see job status and act on it — to be confirmed whether Part 3 covers this or a separate view is needed
  • Process for Bainsy to update or flag overdue part pricing needs to be defined before build
  • Native Tradify to MYOB connection to be verified as active before build
Verification
PrerequisitePart 1 complete and item catalogue populated in Tradify
End to endComplete a test job with catalogue items — draft invoice reaches MYOB with correct line items and pricing
05
Warranty repairs to supplier systems
Per-supplier; approach confirmed with each supplier before scoping
Situation

A handful of suppliers carry most of the warranty volume. Each supplier's actual warranty claim process needs to be confirmed before any scoping. Feasibility and approach will vary per supplier.

Most likely path

Generate a pre-filled claim document (PDF or structured email) from Tradify job data (serial number, fault description, invoice reference, photos) for staff to submit. Direct connection to a supplier system only where the supplier offers one.

Approach
  • One supplier at a time, starting with highest volume
  • Confirm the supplier's warranty process before scoping any build
  • Document generation as the default; direct connection as the exception
